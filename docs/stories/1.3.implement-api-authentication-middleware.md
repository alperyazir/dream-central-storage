# Story 1.3: Implement API Authentication Middleware

## Status
Done

## Story
As a developer, I want to implement a token-based authentication middleware, so that API endpoints can be secured and ready for future role-based access control (RBAC).

## Acceptance Criteria
1. A middleware is implemented that checks for a valid bearer token in the `Authorization` header of incoming requests.
2. Endpoints protected by the middleware return a `401 Unauthorized` or `403 Forbidden` error if a valid token is not provided.
3. The authentication system is designed to be easily extensible for RBAC in the future (e.g., by decoding roles from a JWT).

## Dev Notes
- Security approach per `docs/architecture/security.md`:
  - Auth method is token-based (Bearer). Sensitive endpoints MUST enforce authentication, and future JWT-based roles are expected.
  - While the ACs specify middleware, the architecture also endorses FastAPI `Depends` for protected routes. This story implements middleware for header validation and request context, while providing `get_current_user()` in `app/api/v1/deps.py` to align with Depends usage for future RBAC.
  - Secrets/config must be loaded via centralized config (`app/core/config.py`).
- Source structure per `docs/architecture/source-tree.md`:
  - Add `app/api/v1/deps.py` for auth dependencies, `app/core/config.py` for settings, register middleware in `app/main.py`.
- Suggested environment variables (MVP):
  - `AUTH_BEARER_TOKEN` — static token for MVP authentication
  - Optional for JWT-ready design: `JWT_PUBLIC_KEY`, `JWT_ALG` (not enforced in this story)
- Public endpoints to keep unauthenticated: `GET /health`, `GET /storage/health`.
- Error handling and logging should follow standards in `docs/architecture/error-handling-strategy.md` and general coding standards.

Edge cases and error semantics:
- Missing `Authorization` header → 401 with `WWW-Authenticate: Bearer`.
- Present but malformed header (no scheme or not `Bearer`) → 401.
- `Authorization: Bearer` with empty token → 401.
- Token present but invalid/unauthorized (static mode mismatch) → 403.
- Multiple `Authorization` headers → treat as invalid → 401.
- Public path bypass: requests to `/health` and `/storage/health` must bypass auth; add future `/api/v1/login/token` to bypass when introduced for JWT issuance.

Extensibility and context shape:
- Attach context at `request.state.auth` for future RBAC:
  - Example shape: `{ "sub": null | str, "roles": list[str], "scopes": list[str], "token_type": "static"|"jwt" }`.
- JWT placeholders: if `JWT_PUBLIC_KEY` and `JWT_ALG` are configured, code should be structured to allow easy insertion of JWT verification in a future story, but not required in this one.

### Testing
- Unit tests (pytest) under `app/tests/unit/`:
  - Middleware behavior with missing/invalid/valid tokens
  - Public path bypass
- Integration tests under `app/tests/integration/`:
  - Protect a sample route and verify end-to-end responses with TestClient
- Coverage target 80%+ per `docs/architecture/test-strategy-and-standards.md`.

Recommended concrete scenarios:
- 200 OK when calling a protected test route with `Authorization: Bearer {AUTH_BEARER_TOKEN}`.
- 401 when no `Authorization` header on protected route (check `WWW-Authenticate` header is `Bearer`).
- 401 when scheme is not `Bearer` (e.g., `Basic ...`).
- 401 when `Bearer` token is empty.
- 403 when token provided but does not match configured static token.
- 200 OK for public paths (`/health`, `/storage/health`) without any header.
