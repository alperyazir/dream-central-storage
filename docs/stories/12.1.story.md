# Story 12.1: Security Audit

## Status
Ready for Review

## Story
**As a** system administrator,
**I want** a thorough security audit of the DCS codebase,
**so that** all vulnerabilities are identified and documented before production deployment.

## Acceptance Criteria
1. All API endpoints verified to have proper authentication guards (JWT or API key)
2. API key implementation reviewed: generation randomness, bcrypt hashing, timing-safe comparison, expiry enforcement
3. MinIO bucket policies verified — no public access, presigned URLs properly scoped
4. All storage/file endpoints checked for path traversal vulnerabilities
5. All request schemas validated with Pydantic — no unvalidated user input reaches DB or filesystem
6. File upload limits enforced (size, type)
7. No hardcoded credentials in source code
8. CORS configuration reviewed and documented
9. Webhook HMAC signature validation verified
10. Processing pipeline authorization verified — only authenticated users can trigger
11. Security audit report generated at `docs/qa/security-audit-report.md`

## Dev Notes

### Key Files to Audit
- `apps/api/app/core/security.py` — JWT + API key auth
- `apps/api/app/core/config.py` — Settings and env vars
- `apps/api/app/main.py` — CORS, middleware
- `apps/api/app/routers/*.py` — All endpoint auth guards
- `apps/api/app/services/webhook.py` — Webhook delivery
- `apps/api/app/services/queue/worker.py` — Processing worker
- `infrastructure/docker-compose.yml` — MinIO config, env vars

### Auth Pattern
Every router has a `_require_auth` function that tries JWT first, falls back to API key via `verify_api_key_from_db`. API key auth returns `-1` as user_id to distinguish from JWT users.
