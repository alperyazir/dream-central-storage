# Story 1.4: Upload Application Build

## Status
Ready for Review

## Story
As an administrator, I want to upload a FlowBook application build via a secure API endpoint, so that it can be stored and versioned in the central storage.

## Acceptance Criteria
1. A secure `POST /api/v1/apps` endpoint is created that requires authentication.
2. The endpoint accepts a file upload, a `version` tag, and a `platform` value in {`linux`, `macos`, `windows`}.
3. Upon successful upload, the file is stored in the S3 bucket under a structured path `apps/{platform}/{version}/flowbook.zip` (lowercase segments).
4. A success response is returned with `version`, `platform`, and `location` of the stored file.

### Edge Cases & Assumptions
- `version` is required and non-empty; future stories may enforce stricter semver.
- `platform` must be one of {linux, macos, windows}; case-insensitive inputs are normalized to lowercase.
- If the object already exists, return success with idempotent overwrite or `409 Conflict` based on chosen policy (document chosen behavior in README; default: overwrite for MVP).
- File type is expected to be a zip; content-type/extension enforcement can be advisory in MVP and stricter later.
- Large uploads: streaming upload via `UploadFile` to avoid loading into memory.

### Testing
- Unit tests (pytest) under `app/tests/unit/`:
  - Authorized upload returns 201 with JSON containing `version` and `location`
  - Unauthorized (no header) → 401; wrong token → 403
  - Missing `version` or file → 422
- Integration tests under `app/tests/integration/` (requires MinIO):
  - Upload a small zip to `apps/{version}/flowbook.zip`, then verify existence via MinIO client
- Coverage target: 80%+ per test strategy.
  - [Source: docs/architecture/test-strategy-and-standards.md]
